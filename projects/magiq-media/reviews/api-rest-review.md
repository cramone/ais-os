## Executive Summary

**Overall quality assessment:** The spec is unusually well-documented for a pre-implementation API — every endpoint has request/response examples, explicit error catalogs, and command/event/projection traceability. The shared conventions document (`api-conventions.md`) is genuinely good: RFC 9457 errors, a real idempotency story, a defensible cursor-pagination design, and an explicit versioning/deprecation policy are all things many production APIs never write down. The response-identifier-naming rules (`docs/adrs/api-http-conventions.md §Response Identifier Naming`, formerly ADR-012) in particular show the team already caught and fixed a real class of bug (identifier-naming drift) once — and, as of the 2026-07-08 ADR consolidation, that same document now transparently records the still-open gaps this review found (see Critical Issues C1/C2 below) rather than presenting the rule as fully enforced. That's a good sign for process maturity even though the underlying gaps aren't fixed yet.

The problem is that the response-identifier-naming fix did not fully propagate, and several of the shared conventions that are written down are not actually followed by the individual aggregate specs. This is a **documentation-consistency problem, not a design-philosophy problem** — the team has good taste, but the 11 aggregate specs were evidently written or revised at different times by different passes (possibly different authors or different AI-assisted drafting sessions) without a final cross-endpoint consistency pass. That is exactly what this review is for.

**Overall consistency rating: 5/10.** Individual endpoints are internally coherent and well-specified. Consistency collapses at the cross-endpoint level: list envelopes, status codes for near-identical operations, and filter-parameter names diverge endpoint-to-endpoint in ways that will directly hurt SDK/client-codegen quality and increase caller error rates.

**Major concerns:**

1. The shared pagination contract — `items` / `nextPageToken` / `pageSize`, no `totalCount` (formalized by `docs/adrs/api-http-conventions.md §Cursor-Only Pagination`, formerly ADR-014) — is followed by only 3 of ~11 list endpoints. The rest invent their own envelope key (`collections`, `folders`, `changeRequests`, `assets`, `sessions`, `jobs`) and omit `pageSize`.
2. The response-identifier-naming rules ("own identifier is always `Id`," `docs/adrs/api-http-conventions.md §Response Identifier Naming`) are violated by every bulk-import-job endpoint (`jobId` instead of `id`), and the rules' own Rule 4 worked example (`CollectionId` in bulk sub-items) is contradicted by every bulk-create endpoint actually shipping `id` instead.
3. `202 Accepted` is used inconsistently for what the spec's own async-operations table defines as synchronous, non-saga operations (Asset archive, MediaProfile deprecate, RecordType deprecate), while functionally identical operations elsewhere (Collection archive, Folder archive) correctly return `204`.
4. The same logical operation — "filter a list by the owning MediaItem" — uses three different query parameter names across endpoints (`itemId`, `mediaItemId`, and implicit via path).
5. Bounded-context URL prefixing was applied to 2 of 6 contexts (`/catalog/...`, `/metadata/...`) and dropped entirely for the other 4 (`/assets`, `/change-requests`, `/signing/...`, `/registrations` all live at root). **Decided 2026-07-08:** the platform is standardizing on flat, resource-oriented URLs for all six contexts — a DDD module boundary is an internal-architecture concern and shouldn't be coupled to the public route, since that coupling turns a future module refactor into a breaking API change. See the URL Naming section below; execution (dropping the `/catalog`, `/metadata`, and `signing` segments) is tracked in the remediation plan, Stage 5.

**Minor concerns:** inconsistent use of `200 OK` with no body vs. `204 No Content`; inconsistent response body presence on `201 Created`; a metadata full-replace endpoint on MediaItem breaks its own array/map shape from a prior version without a migration path; several literal string-corruption artifacts in the spec text (`"media-assets"`, `"media-items"` used as JSON keys, doubled `media-` prefixes) that read as copy/paste or find-replace errors and should be fixed regardless of whether they're intentional.

**Positive observations:** the RFC 9457 error contract is applied uniformly and is genuinely good; idempotency-key handling is spelled out once and referenced everywhere rather than redefined per endpoint; the bulk-operation partial-success envelope (`succeeded`/`failed`/`skipped`, `onError`, `onDuplicate`) is a strong, consistently-described pattern; the nested-creation/flat-operation route pattern (ADR-documented) is a defensible, explicitly-justified exception to strict REST purity rather than an accidental inconsistency; the folder assignment/move decision (`docs/adrs/api-http-conventions.md §Folder Assignment and Move`, formerly ADR-011) collapsing `PUT .../folder` + `POST .../move` into a single idempotent `PUT` is exactly the right call and is well-reasoned; and the cursor-only pagination decision (`§Cursor-Only Pagination`, formerly ADR-014) is a clean, well-justified resolution of what this review originally flagged as part of Critical Issue C4 — see Critical Issues below.

## Critical Issues

These should be resolved before implementation locks in the wire contracts (some are already flagged `🔧 Requires implementation` in the spec, meaning code doesn't exist yet — cheapest possible time to fix).

| # | Issue | Why it's critical |
|---|---|---|
| C1 | **Response-identifier-naming rules (`docs/adrs/api-http-conventions.md §Response Identifier Naming`) violated by every `BulkFolderImportJob`/`BulkMediaImportJob` endpoint** — job resources expose their own identifier as `jobId`, not `id`. | Those rules were written specifically to eliminate this pattern platform-wide and state all existing contracts were remediated. This is a clear regression/gap — either the bulk-import-job specs were written after the remediation sweep and never brought into line, or the sweep missed a context. Generic client code (`resource.id`) will silently fail for this one aggregate. As of the 2026-07-08 ADR consolidation, this gap is now acknowledged inline in `api-http-conventions.md` itself rather than being a silent contradiction — still open, but now tracked at the source. |
| C2 | **Rule 4's own worked example is contradicted by every bulk-create endpoint.** The rule specifies `BulkCreateCollectionsSucceededModel(int Index, string CollectionId, string Name)`, but `collection.api.md`, `folder.api.md`, `mediaitem.api.md`, and `asset.api.md` all show `"id"` (not `CollectionId`/`FolderId`/`ItemId`/`AssetId`) in their `succeeded[]` arrays. Meanwhile `POST /v1/catalog/items/bulk/metadata` uses `"itemId"` in its own `succeeded[]` array — a third convention in the same aggregate. | This is not a minor nit — it means the one part of the spec written specifically to prevent identifier-naming drift is itself internally contradictory, and three different conventions exist for "identify this item inside a bulk array" within the same context. Pick one before any client SDK or Postman collection is generated from this spec. Also now acknowledged inline in `api-http-conventions.md`. |
| C3 | **`202 Accepted` used for synchronous, non-saga operations**, contradicting the spec's own definition of when `202` applies. `api-conventions.md §Async Operations` states `202` is for "endpoints that initiate asynchronous media-sagas" and lists exactly two saga-triggering endpoints (`POST .../items/{id}/submit`, `POST .../items/{id}/signing-sessions`). Yet `POST /v1/assets/{id}/archive`, `POST /v1/catalog/profiles/{id}/deprecate`, and `POST /v1/metadata/record-types/{id}/deprecate` all document `202 Accepted` despite their own traceability tables showing a single synchronous command → event, no saga. Meanwhile `POST /v1/catalog/collections/{id}/archive` and `POST /v1/catalog/folders/{id}/archive` — semantically the same "archive" action — correctly return `204`. | Callers cannot distinguish "this will complete asynchronously, poll for it" from "already done" without reading the aggregate-specific narrative text, defeating the entire purpose of having a documented status-code convention. This will produce either unnecessary client polling loops or race conditions where clients assume completion prematurely. |

**Resolved since the initial pass:** this review originally carried a fourth critical issue (C4) — the pagination response contract wasn't consistently implemented, including a `totalCount` field the shared spec mandated but no endpoint actually returned. `docs/adrs/api-http-conventions.md §Cursor-Only Pagination` (formerly ADR-014) resolved the `totalCount` half by formally removing it from the contract — cursor-based DynamoDB/OpenSearch pagination can't produce a total cheaply, so no endpoint needed to change; the shared doc was the thing out of step with practice, and that's fixed. The residual issue — inconsistent list-envelope *key names* and inconsistent presence of `pageSize` — was never Critical-severity on its own (it was already tracked as High under Pagination Issues) and remains open there; see that section below.

## REST Design Issues

### URL Naming Issues

| Severity | Endpoint(s) | Current | Recommended | Reasoning |
|---|---|---|---|---|
| High | All contexts | Bounded-context prefixing is applied inconsistently: `Catalog` → `/v1/catalog/...`, `Metadata` → `/v1/metadata/...`, but `AssetManagement` → `/v1/assets` (root), `ChangeRequests` → `/v1/change-requests` (root), `DocumentSigning` → `/v1/signing/...` (root, abbreviated context name), `Registration` → `/v1/registrations` (root). | **Decided 2026-07-08:** drop the prefix from Catalog and Metadata — flat, resource-oriented URLs platform-wide (`/v1/collections`, `/v1/folders`, `/v1/items`, `/v1/profiles`, `/v1/record-types`, `/v1/sessions`, etc.), matching the majority (4 of 6) contexts that already do this. Grouping for documentation/discoverability purposes (e.g. "show me everything Catalog owns" in Swagger) should be handled with OpenAPI tags, not URL segments. | Two contexts prefixing and four not is not "some resources are top-level, others are nested" (a legitimate REST distinction) — it's an unexplained mix of the *same* kind of decision (context grouping) applied two different ways. A client building a mental model of "where do I find X" cannot infer the rule from the pattern as it stands. A URL should model the resource graph as the caller sees it, not the internal DDD module layout — module boundaries can and do shift independently of what a resource *is*, and baking them into the route couples the public contract to an implementation detail that's expected to change. Execution tracked in the remediation plan, Stage 5. |
| Medium | `DocumentSigning` routes | Uses `/v1/signing/sessions/...` — "signing" is an abbreviation of the bounded context name `DocumentSigning`, whereas every other context that does prefix (`catalog`, `metadata`) uses the literal (lowercased) context name. | `/v1/sessions/...` — under the confirmed flat-URL decision above, drop the `signing` segment entirely rather than expanding it to `document-signing`. There's no context prefix left to be consistent or inconsistent with once every context is flat. | Was a second axis of unpredictability on top of the prefix-or-not question; moot now that the prefix-or-not question is resolved in favor of "not." |
| Medium | `mediaprofile.api.md` traceability table | Lists `POST /v1/metadata/record-types/{id}` and `PUT /v1/metadata/record-types/{id}/version` for attaching/updating a RecordType reference on a MediaProfile — but the route table two sections above in the *same file* correctly gives `POST /v1/catalog/profiles/{profileId}/record-types/{recordTypeId}` and `PUT /v1/catalog/profiles/{profileId}/record-types/{recordTypeId}/version`. | Fix the traceability table to match the route table. | This is an internal, same-document contradiction — not a cross-file consistency question. It reads as a copy/paste error from the RecordType aggregate's own API doc and should be a trivial fix, but as written it would mislead anyone generating client code or tests from the traceability table instead of the route table. |
| Medium | MediaItem resource segment | Aggregate is `MediaItem`, URL resource is `items` (`/v1/catalog/items`). Response bodies, error codes, and prose consistently say "MediaItem" / "media item". Bulk endpoints even use the literal string `"media-items"` as a JSON array key in one place (see Response Model Issues). | Keep `items` (it's consistent with the platform's practice of dropping the `Media` prefix in URLs — see `profiles`, `change-requests` — this is not itself a problem), but fix the literal `"media-items"` JSON key artifacts described below so the URL convention and the body convention don't visibly contradict each other in the same document. | The URL choice itself is fine and consistent with the rest of the platform's prefix-dropping convention. The issue is the accidental literal `media-items` string leaking into JSON examples, which looks like a stray, unintentional artifact rather than a deliberate naming choice. |
| Low | Registration MediaItem-document sub-resource | `POST /v1/registrations/{registrationId}/documents` — "documents" here actually means "an attached MediaItem playing the role of a supporting document," per the request body (`mediaItemId`, `itemType`). | Consider `POST /v1/registrations/{registrationId}/attached-items` or keep `documents` but note the terminology divergence in the spec (MediaItem vs. Document vs. Asset are all distinct concepts elsewhere in the platform — see Naming Consistency Issues). | Not wrong, but worth a second look given the platform elsewhere treats "media item," "asset," and "document" as three distinct, non-interchangeable domain terms (Asset has its own dedicated `media-documents` S3 bucket for non-processable content). Using "document" as a generic sub-resource name here risks confusion with that more specific meaning. |
| Low | `bulk-operations.md` re-submission example | Shows `POST /media-collections/bulk` in a worked example — no `/v1` prefix, no `/catalog` segment, and the resource name `media-collections` doesn't match the actual route (`/v1/catalog/collections/bulk`). | Fix the example to `POST /v1/catalog/collections/bulk`. | Minor, but this is exactly the kind of copy-pasted example that ends up in a Postman collection or onboarding doc verbatim. |

## Recommended Standards

Consolidating the above into a single set of conventions to adopt platform-wide, before implementation locks these contracts in:

**Resource identifiers.** Every resource's own primary key is `id` in every response, everywhere, with no exceptions other than the two the response-identifier-naming rules already grant (`docs/adrs/api-http-conventions.md §Response Identifier Naming`, formerly ADR-012: version sub-resources keyed by parent + version number; integration-event/cross-context transport contracts). This means fixing `jobId` → `id` on both bulk-import-job aggregates, and resolving the three-way split (`id` / `CollectionId`-style / `itemId`) on bulk-array sub-items by either updating Rule 4 to match the `id` convention already used by 4 of 5 bulk endpoints, or updating those 4 endpoints to match the rule. Given the weight of existing (if inconsistent) practice, standardizing on `id` in bulk arrays too — and simplifying Rule 4 to say "same as Rule 1" — is the lower-disruption path, since bulk-array entries are always accessed via `.succeeded[i].id`-style code that's already written expecting `id` in most places.

**List responses.** Every paginated list response uses the same envelope: `{ items: T[], nextPageToken: string | null, pageSize: number }` for DynamoDB-backed lists, or the OpenSearch equivalent (`items`, `nextSearchAfter`, `pageSize`) already correctly documented for search endpoints. **No `totalCount` / `totalPages` / `pageCount` field**, per `docs/adrs/api-http-conventions.md §Cursor-Only Pagination` (formerly ADR-014) — cursor-based pagination against DynamoDB and OpenSearch doesn't make that number cheap to produce, and the platform has now formally ruled it out rather than leaving it as a permanently-`null` placeholder. No type-specific envelope keys (`collections`, `folders`, etc.) either — this is exactly what the shared convention already specifies; the fix is to bring the 7 non-compliant endpoints into line with it, not to invent something new.

**Query parameters for filtering.** Add a Filtering section to `api-conventions.md`, modeled on the existing Sorting section, with one row per canonical filter relationship used anywhere in the platform. At minimum: `ownerId`, `mediaItemId` (never `itemId`), `collectionId`, `folderId`, `status`, `unassigned`. Every new endpoint consults this table instead of choosing a name locally.

**HTTP status codes for mutations.**
- `201 Created` + body containing at least `{ id }` (ideally + `Location` header) for every resource-creation endpoint, no exceptions (fixes the comment-creation gap).
- `204 No Content` for every "succeeded, nothing to return" mutation. Never `200 OK` with an empty body.
- `202 Accepted` reserved exclusively for the two documented saga-triggering endpoints (or any future ones added to that same table in `api-conventions.md`) — never used as a generic "something async-ish happened" status for synchronous single-command operations. If an endpoint's traceability table shows one command → one event with no saga, it does not get `202`.
- `PUT`, not `POST`, for any endpoint whose own documentation says "full replace" / "set to exactly this value" — this already correctly describes the platform's practice for `Folder.parent`, `RecordType` field replacement, and MediaProfile policy setters; extend it to the three tag-replacement endpoints.

**Bounded-context URL structure.** **Decided 2026-07-08:** flat, resource-oriented URLs platform-wide — no bounded-context prefix segments. Remove `/catalog` from Collection/Folder/MediaItem/MediaProfile routes and `/metadata` from RecordType routes; drop the `signing` abbreviation from DocumentSigningSession routes too, since it was a partial context-prefix artifact rather than a deliberate resource name. Document the policy explicitly in a new `api-conventions.md §URL Structure` section so future aggregates don't reintroduce prefixing. Execution tracked in the remediation plan, Stage 5.

**JSON key hygiene.** A one-time find-pass for `media-` prefixes leaking into JSON code-fence keys (`"media-assets"`, `"media-items"`) and doubled `media media-` prose artifacts, since these read as unintentional corruption rather than a deliberate — if unusual — naming choice.

## Overall Assessment

| Dimension | Score | Notes |
|---|---|---|
| REST maturity | 7/10 | Individual endpoints show strong REST fundamentals: correct verb choice in the majority case, RFC 9457 errors, real idempotency semantics, a defensible pagination design, and at least one decision (the folder assignment/move rule — `docs/adrs/api-http-conventions.md §Folder Assignment and Move`, formerly ADR-011) that gets a genuinely subtle REST question (idempotent PUT vs. RPC-style POST) exactly right with clear reasoning. Docked points for the `202`-on-synchronous-operations pattern and the tags-should-be-PUT issue, both of which contradict reasoning the platform has *already* correctly applied elsewhere in the same spec. |
| Consistency | 4/10 | This is the weak dimension. The number of "two or three different conventions exist for the identical structural problem, in the same platform, sometimes in the same file" findings (list envelopes, bulk-array identifiers, filter param names, 200-vs-204) is high enough that a client SDK generated naively from this spec would need substantial per-endpoint special-casing. |
| Maintainability | 6/10 | The traceability tables (Command → Event → Projection) present in every aggregate spec are a genuinely excellent maintainability feature — few teams document this level of causal chain per endpoint. The error catalog is centralized and cross-referenced well. What drags this down is that the *shared* conventions documents (`api-conventions.md`, the response-identifier-naming rules in `docs/adrs/api-http-conventions.md`) are demonstrably not being kept in sync with the individual aggregate specs as those get revised — meaning the shared docs currently function more as aspirational targets than as an enforced contract. Fixing the drift once and then treating the shared docs as a gate for future endpoint additions (e.g., a lightweight PR checklist referencing this review) would move this to an 8+. |

**Recommended priority of fixes**, roughly in the order they should land relative to implementation:

1. **Before any code is written against these contracts:** resolve the `202`-vs-`204` question (C3) and the `200`-vs-`204` empty-body question (HTTP Status Code Issues, High) — these are pure decisions with no data-shape consequences, cheapest to fix now, and touch the largest number of endpoints.
2. **Before any client SDK or Postman collection is generated:** resolve the identifier-naming three-way split (C1/C2) and the list-envelope key/`pageSize` inconsistency (Pagination Issues, High) — these have real data-shape consequences and get more expensive the longer real client code exists against the current (inconsistent) shapes.
3. **Before the MediaItem metadata endpoints ship:** confirm the metadata collision-prevention decision's (`docs/adrs/catalog-domain-invariants.md §Metadata Collision Prevention and General Fields`, formerly ADR-013) breaking-change premise (no released version yet) still holds, or route it through `/v2` per the platform's own versioning policy.
4. **Lower urgency, can land alongside normal spec maintenance:** the `POST`-vs-`PUT` tags fix, executing the now-confirmed flat-URL policy (dropping the `/catalog`, `/metadata`, and `signing` segments — decision made 2026-07-08, only the mechanical route-table edits remain), the filter-parameter table addition to `api-conventions.md`, and the JSON-key/prose corruption cleanup.

None of these require re-architecting anything — every fix in this review is a naming, status-code, or response-shape correction within the existing route and domain-event structure. The underlying domain model, command/event design, and bulk-operation semantics are sound; this is a consistency pass, not a redesign.
